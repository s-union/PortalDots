# Architecture

## Overview

PortalDots is a split-stack web application: a Vue 3 SPA served from a CDN-friendly static build, talking to a Go API server that owns all persistent state.

```
Browser
  └── Vue SPA (static files)
        └── /v1/* (proxied in dev, deployed separately in prod)
              └── Go API server
                    └── PostgreSQL
```

Email delivery is separated from the request path entirely and runs through a Cloudflare Workers Queue pipeline.

---

## Why a split stack?

The two halves of the application have fundamentally different operational profiles:

- **Frontend**: changes frequently, needs hot-reload in dev, benefits from CDN edge caching in production.
- **Backend**: owns transactions, authentication, file storage, and email delivery. Stability matters more than iteration speed.

Keeping them as separate processes lets each evolve independently and be deployed to appropriate infrastructure (static host + cheap VPS or container).

---

## API contract as the source of truth

`backend/api/openapi.yaml` is the single definition of the HTTP interface between the two halves. Neither side is the source of truth for field names or types — the spec is.

The TypeScript client in `packages/api-client/` is generated from that spec via `openapi-typescript`. Any drift between the spec and the implementation surfaces immediately as a type error in either the frontend consumer or the Go handler.

When the shape of an endpoint changes:
1. Update `backend/api/openapi.yaml`.
2. Run `mise run frontend-generate`.
3. Fix any TypeScript errors before finishing.

This forces changes to be explicit and bi-directional rather than discovered at runtime.

## Request lifecycle

```
Browser
  → Vue Router (client-side)
  → TanStack Query (cache + fetch)
  → openapi-fetch (typed HTTP client)
  → Go Echo server
      → rate-limit middleware
      → external-ID decode middleware
      → session middleware
      → handler (reads domain interfaces)
      → domain repository (calls sqlc-generated SQL)
      → PostgreSQL
  ← JSON response (external IDs re-encoded)
```

### External ID obfuscation

PostgreSQL primary keys are UUIDs. The API never exposes raw UUIDs to the browser. Instead, a middleware layer (`internal/middlewares/external_ids.go`) encodes UUIDs to a compact opaque string on the way out and decodes them back on the way in. This prevents clients from constructing valid IDs by guessing, and decouples the public ID surface from the internal key space.

## Authentication and session model

Sessions are stored in PostgreSQL (`sessions` table), not in memory or a separate cache. This means:

- Sessions survive server restarts.
- Multiple server replicas share the same session state without extra infrastructure.
- Revoking a session (logout, role change, security action) is a simple `DELETE` — no cache invalidation needed.

Session cookies use `HttpOnly; SameSite=Strict` and are scoped to the API server origin. The cookie name and TTL are configurable via `PORTAL_SESSION_COOKIE` and `PORTAL_SESSION_TTL_SECONDS`.

---

## Staff verification

Staff-mode access requires a second verification step beyond the normal session. The staff verify code (`PORTAL_STAFF_VERIFY_CODE`) is an out-of-band secret set by an administrator — not a per-user credential. This deliberately limits the blast radius of a compromised session: knowing the session cookie is not enough to reach staff-only endpoints.

In `PORTAL_DANGEROUSLY_ALLOW_DEMO_MODE=true`, this step is bypassed automatically so that local development does not require the out-of-band flow.

## Data model

The core entities and their relationships:

```
users ─────────────────── circle_user ─── circles
  │                                           │
  │                               participation_types
  │
user_login_ids   (multi-identity: student ID, email, etc.)
user_roles       (admin, staff)
user_permissions (fine-grained staff capabilities)

circles ─── forms ─── form_questions
               │
               └── answers ─── answer_details
                          └── answer_uploads

pages ─── documents   (announcements + file attachments)
pages ─── tags        (viewability filter)

mail_queue            (outbound mail jobs, delivered async)
activity_logs         (audit trail for all staff mutations)
sessions              (server-side session storage)
pending_registrations (email-verification flow state)
```

All primary keys are `uuid` generated by `uuidv7()` — a PostgreSQL extension function installed in the first migration. UUIDv7 is time-ordered, so index insertions are sequential and there is no hot-spot fragmentation.

---

## Dependency injection

The server wires everything explicitly in `main.go`. Domain interfaces (e.g., `circle.Catalog`, `form.Repository`) are satisfied by sqlc-backed implementations built in `internal/platform/database/`. No global state, no service locator. The composition root is the only place that knows the concrete types.

This makes it straightforward to substitute a static in-memory implementation (used for demo mode and tests) without touching any handler code.
